This file provides guidance to Claude Code when working with code in this repository.

## Repository Purpose

This repo contains Helm chart definitions for Union's selfhosted/selfmanaged deployments. It
does **not** stand alone — it is consumed by the `unionai/cloud` repo as part of a GitOps pipeline.

## Charts

- `charts/controlplane/` — Union control plane (flyteadmin, identity, executions, etc.)
- `charts/dataplane/` — Union dataplane (union-operator / propeller)
- `charts/dataplane-crds/` — CRDs for the dataplane chart
- `charts/knative-operator/` — Knative operator
- `charts/sandbox/` — Sandbox / all-in-one deployment

## Values File Layering

Values files are layered by the deployment tooling, not by Helm's `-f` chaining alone.
Understanding the naming convention is critical:

| Pattern | Purpose |
|---------|---------|
| `values.yaml` | Base defaults for all deployments |
| `values.{cloud}.yaml` | Platform defaults (e.g. `values.gcp.yaml`, `values.aws.yaml`) |
| `values.{cloud}.{mode}.yaml` | Platform + deployment mode (e.g. `values.gcp.selfhosted-intracluster.yaml`) |
| _(env-specific)_ | Generated by Terraform in `unionai/cloud`, lives in that repo |

The base and platform/mode values files in this repo provide **defaults**. They are fetched
via HTTP from GitHub raw URLs by `union_extension/{cloud}/controlplane.tf` (and `dataplane.tf`)
in the cloud repo, then merged with environment-specific overrides and written as
`envs/<env>/<app>/values.yaml` in the cloud repo. ArgoCD reads both repos to deploy.

**Do not put environment-specific values (domains, IPs, credentials) here.** Those belong in
the generated overlay in the cloud repo.

## GitOps Deployment Path

```
helm-charts (templates + platform defaults)
    +
cloud/infra/terraform/envs/<env>/control-plane/values.yaml  (Terraform-generated, env-specific)
    |
    v
ArgoCD ApplicationSets (in cloud/infra/argocd/deploy/manifests/)
    |
    v
Kubernetes cluster
```

ArgoCD's ApplicationSets (`appset-selfmanaged-control-plane.yaml`, etc.) specify both repos as
Helm sources and merge them at deploy time. The revision of this repo to use is pinned per
environment in `revisions.yaml` files in the cloud repo.

## Testing

Snapshot tests live in `tests/`. After changing templates or values files, regenerate snapshots and run tests:

```bash
# From repo root
make generate-expected  # Regenerate snapshot files
make test               # Run comparison tests against snapshots
helm template ...       # Manually inspect rendered output
```

All tests must pass before merging. Generated test snapshots (`tests/generated/*.yaml`) should
be committed alongside template changes.

## Metrics Glossary Maintenance

When modifying any of these files, the metrics glossary may need updating:

- `charts/controlplane/dashboards/*.json` — Grafana dashboard panels
- `charts/dataplane/dashboards/*.json` — Grafana dashboard panels
- `charts/*/templates/monitoring/prometheusrule.yaml` — Recording rules and alerts

After changes, regenerate the metrics manifest and commit it:

```bash
make generate-metrics-manifest
```

If the manifest diff shows added, removed, or renamed metrics, the customer-facing
metrics glossary in `unionai-docs` must also be updated:

- `unionai-docs/content/deployment/selfhosted/monitoring/metrics-glossary.md`

The manifest (`metrics-manifest.yaml`) is auto-generated — do not edit it manually.
